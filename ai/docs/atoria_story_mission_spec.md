# Atoria Story/Mission Spec v0.1

## 목적

이 문서는 아토리아의 스토리/미션 생성 품질 기준을 정의한다.

프롬프트, RAG 사용 방식, validator, golden sample은 이 문서를 기준으로 맞춘다. 목표는 문화재 설명문을 만드는 것이 아니라, 사용자가 현장에서 직접 참여하고 사진 인증을 통해 경험을 남기는 동화형 퀘스트를 만드는 것이다.

## 기획 기준

아토리아의 핵심 문장:

- 여행이 동화가 되는 순간
- 사용자는 단순 관람객이 아니라 이야기의 주인공
- 문화재 현장에서 미션을 수행하고, 그 여정을 AI가 맞춤형 스토리와 E-book으로 남긴다
- 사진, 행동, 선택 기반 인터랙션을 통해 사용자가 능동적으로 참여한다

따라서 생성 결과는 다음 방향을 따른다.

- 문화재를 설명하는 콘텐츠가 아니라 문화재를 무대로 한 체험 콘텐츠여야 한다.
- 사용자가 현장에서 어떤 행동을 해야 하는지 분명해야 한다.
- 사진은 미션 뒤에 붙는 부가 행동이 아니라 미션 클리어 증거여야 한다.
- 나중에 E-book에 들어갔을 때 "내가 이 장면을 해결했다"는 기억으로 읽혀야 한다.

## 생성 결과의 기본 구조

스토리 생성 결과는 다음 단위를 가진다.

```text
intro
places[]
  - story_fragment
  - mission_title
  - mission_instruction
  - verification_hint
outro
```

각 장소는 독립 설명문이 아니라 하나의 연결된 여정 안의 장면이어야 한다.

## Story Fragment 규격

`story_fragment`는 장소 해설이 아니라 동화 속 장면이다.

필수 요소:

1. 현재 장소 전체명
2. RAG 기반 역사 사실, 설화, 상징 중 최소 1개
3. 사용자가 그 장소에서 맡는 역할 또는 감정
4. 다음 장소나 미션으로 이어지는 단서

좋은 구조:

```text
현재 장소 전체명에서 앱 속 이야기 장치나 단서가 나타난다.
RAG 기반 소재가 장면 안에 자연스럽게 섞인다.
사용자가 그 단서를 해결해야 다음 장소로 갈 수 있다.
```

나쁜 구조:

```text
이 문화재는 신라 시대의 문화재입니다.
이곳에서는 역사적 의미를 느낄 수 있습니다.
사진을 찍어보세요.
```

문체 기준:

- 초등 고학년이 읽을 수 있는 쉬운 한국어
- 동화처럼 부드럽지만 과하게 유치하지 않음
- 해설사 말투보다 장면을 보여주는 말투
- "도착하면", "보입니다", "느껴보세요" 반복 지양
- 사용자를 관람객이 아니라 발견자, 수호자, 기록자, 별을 읽는 아이 등으로 위치시킴

## Mission 규격

아토리아의 미션은 게임형 퀘스트다.

미션은 항상 다음 네 요소를 포함한다.

```text
1. 퀘스트 목표: 무엇을 해결해야 하는가
2. 현장 행동: 사용자가 실제로 무엇을 해야 하는가
3. 사진 클리어 조건: 사진 안에 무엇이 보여야 성공인가
4. 이야기 보상: 이 사진이 어떤 단서, 열쇠, 표식, 기억이 되는가
```

사진은 미션 완료 조건의 일부다.

나쁜 예:

```text
불국사 다보탑의 난간과 계단을 사진에 담아보세요.
```

이 문장은 미션이 아니라 촬영 지시다.

좋은 예:

```text
불국사 다보탑에서 보물탑의 문장을 완성해요. 난간과 계단이 한 화면에 들어오도록 자리를 잡고, 두 요소가 함께 보이면 클리어예요. 그 사진이 석가탑으로 이어지는 다음 표식이 됩니다.
```

## Mission Type 규격

현재 내부 타입은 `observation`, `photo`, `quiz`, `imagination`, `route`를 사용한다.

서비스 관점에서는 모든 미션에 사진 인증이 붙을 수 있다. 따라서 `photo` 타입은 "사진만 찍는 미션"이 아니라 "프레이밍 자체가 핵심인 미션"일 때 사용한다.

권장 미션 패턴:

| 패턴 | 사용자 행동 | 사진 클리어 조건 예 | 적합한 장소 |
|---|---|---|---|
| 찾기형 | 숨은 표식, 창, 조각, 문양 찾기 | 표식과 문화재가 함께 보임 | 탑, 창, 문양 |
| 세기형 | 계단, 층, 기둥 수 세기 | 셀 대상이 한 화면에 보임 | 백운교, 석가탑 |
| 맞추기형 | 두 요소를 한 프레임에 맞춤 | A와 B가 동시에 보임 | 대웅전, 탑 배치 |
| 비교형 | 두 구조의 차이 찾기 | 비교 대상이 함께 보임 | 다보탑/석가탑 |
| 경로형 | 특정 순서로 이동 | 출발/도착 단서가 보임 | 계단, 다리 |
| 역할형 | 수호자, 기록자, 별 읽는 아이 역할 수행 | 역할을 보여주는 포즈나 대상 | 가족/아동 |
| 선택형 | 어떤 단서가 맞는지 고름 | 선택한 대상이 보임 | 여러 요소가 있는 장소 |
| 상상형 | 설화/상징을 바탕으로 장면 구성 | 상상 근거가 되는 실제 대상 | 설화/상징 자료가 있는 장소 |

## Verification Hint 규격

`verification_hint`는 "무엇이 사진에 보여야 완료인가"만 적는다.

좋은 예:

```text
불국사 백운교의 18단 계단 흐름과 돌난간이 함께 보이면 클리어.
```

나쁜 예:

```text
사진을 찍으면 클리어.
멋진 모습이 보이면 클리어.
문화재가 보이면 클리어.
```

현재 백엔드는 AI wire 응답의 `verification_hint`를 저장하고, 챕터 상세 응답의 `mission.verificationHint`로 내려준다. 구버전 데이터나 누락 응답을 대비해 `mission_instruction` 안에도 클리어 조건을 유지한다.

## RAG 사용 규칙

RAG는 상상력을 제한하는 자료가 아니라, 장면과 미션을 장소에 묶어 주는 근거다.

사용 원칙:

- `fact_context`: 역사적 사실, 구조, 연도, 기능 설명의 근거
- `legend_material`: 전승, 설화, 인물 이야기 소재. "전해진다", "이야기된다"로 완화
- `symbolic_material`: 상징 해석 소재. "상징적으로 보면", "의미로 해석하면"으로 완화

RAG에서 가져온 사실은 설명문으로 길게 나열하지 않는다. 장면, 단서, 미션 목표 안에 넣는다.

나쁜 예:

```text
첨성대는 신라의 천문 관측 시설입니다.
```

좋은 예:

```text
첨성대의 남쪽 창이 보이자, 준영은 별을 읽던 신라 사람들의 자리 앞에 선 것 같았다.
```

## 안전/접근 제한 규칙

미션은 현장에서 안전하고 실제로 수행 가능해야 한다.

금지:

- 문화재 접촉, 등반, 훼손 가능 행동
- 울타리 안, 비공개 구역, 출입 제한 구역 진입
- 촬영 금지 가능성이 있는 내부 촬영 단정
- 사찰 법당 내부, 불단, 불상 촬영 지시
- 석빙고 내부 진입 지시

장소별 주의:

| 장소 | 제한 |
|---|---|
| 불국사 대웅전 | 내부, 불단, 불상 촬영 지시 금지. 외부 전면, 기둥, 처마, 현판, 앞마당, 주변 탑 배치 중심 |
| 석빙고 | 내부 진입 금지. 내부, 천장, 배수로, 바닥 중앙 언급 금지. 입구, 석축, 지붕선, 안내판 중심 |
| 문화재 일반 | 만지기, 올라가기, 기대기, 플래시 촬영 지시 금지 |

## 금지 표현

아래 표현은 결과 품질을 떨어뜨리므로 피한다.

- 사진에 담아보세요
- 사진으로 남겨보세요
- 그 모습을 사진으로 찍어보세요
- 관찰해보세요만으로 끝나는 문장
- 느껴보세요만으로 끝나는 문장
- 문화재를 둘러보세요
- 이 문화재는 ~입니다
- 역사적 의미를 이해할 수 있습니다
- 도움이 됩니다
- 경험할 수 있습니다

완전 금지가 어려운 경우에도 한 미션의 핵심 문장으로 쓰면 안 된다.

## 권장 표현

- 한 화면에 들어오면 클리어
- 두 단서가 함께 보이면 성공
- 사진 속에 A와 B가 같이 보이면 다음 표식이 열린다
- A를 찾고 B와 맞춰 클리어
- 이 장면이 다음 장소로 가는 열쇠가 된다
- 이 사진이 여행책의 한 페이지가 된다

## 품질 평가표

| 항목 | 통과 기준 |
|---|---|
| 동화성 | 사용자가 이야기 속 주인공으로 행동한다 |
| 미션성 | 단순 촬영/관찰이 아니라 해결 목표가 있다 |
| 사진 인증성 | 사진이 미션 성공의 증거로 작동한다 |
| 장소성 | 다른 문화재에 붙여도 되는 일반 문장이 아니다 |
| RAG 충실도 | 사실/설화/상징을 왜곡하지 않는다 |
| 안전성 | 내부 진입, 접촉, 촬영 금지 가능 행동이 없다 |
| 문체 | 초등 고학년에게 자연스럽고 부드럽다 |
| E-book 활용성 | 사진과 문장이 나중에 여행 이야기로 남기 좋다 |

## Validator/Repairer 후보 규칙

향후 코드 검증에서 강제할 수 있는 항목:

- `사진에 담아보세요`, `사진으로 남겨보세요`가 mission_instruction에 있으면 warning 또는 error
- mission_instruction에 `클리어`, `한 화면`, `단서`, `열쇠`, `표식` 중 하나도 없으면 error
- mission_instruction이 단순 촬영 지시로 끝나면 error
- story_fragment가 `이 문화재는`, `도착하면`, `볼 수 있습니다` 위주면 warning
- 석빙고 문맥에서 `내부`, `천장`, `배수로`, `바닥 중앙`이 나오면 error
- 대웅전 문맥에서 `불단`, `불상`, `내부 촬영`이 나오면 error
- 현재 장소 전체명이 story_fragment 첫 2문장 안에 없으면 warning

## 작업 흐름

1. 이 규격을 기준으로 골든 샘플을 만든다.
2. 프롬프트를 규격 순서에 맞춰 재구성한다.
3. validator/repairer에 강제 규칙을 추가한다.
4. 박준영 케이스로 회귀 검증한다.
5. 여러 성향/코스 테스트 세트로 확장한다.
